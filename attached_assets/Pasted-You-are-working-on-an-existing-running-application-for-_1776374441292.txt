## CHANGE 1 — UPGRADE THE PRE-ANALYSIS (pre-analyze-upload)

File: the route handler for POST /api/products/pre-analyze-upload

### Current behavior (do not break):
- Extracts text via PyMuPDF get_text("dict") with a 20k char stratified sampler
- Sends text to GPT-4o to identify products (ticker, name, product_type, gestora, cnpj)
- Does cascade match against products table (5 levels: exact ticker, ILIKE, prefix, name, aliases)
- Creates new products if needed
- Saves result to Material.ai_product_analysis
- Returns product list with match confidence

### What to ADD (do not remove existing logic):

**1a. Deep key_info extraction per identified product**

After the existing product identification step, make a second GPT-4o call with 
this purpose: for each product already identified, extract structured information.

Use this prompt structure for the second call:
- System: "You are a financial document analyst specializing in Brazilian variable 
  income products. Extract detailed structured information."
- User: Provide the same extracted text + the list of identified products, and ask:
  "For each of the following products identified in this document: {product_list}
   Extract all available information you can find. For each product return a JSON 
   object with these fields (use null if not found):
   {
     ticker: string,
     investment_thesis: string (the main investment rationale, up to 3 sentences),
     expected_return: string (ex: 'IPCA + 6% a.a.', '12% a.a.', 'DY ~11%'),
     investment_term: string (ex: '24 meses', 'longo prazo', 'sem vencimento'),
     main_risk: string (primary risk factor),
     issuer_or_manager: string,
     rating: string or null,
     minimum_investment: string or null,
     liquidity: string or null,
     additional_highlights: array of strings (up to 3 key bullet points)
   }
   If a field truly does not appear in the document, return null for that field.
   Return ONLY a valid JSON array."

Store this extracted info in the existing Product.key_info JSON field when creating 
or updating a product. When updating an existing product's key_info, MERGE the new 
fields — do not overwrite existing key_info entirely; only fill null/missing fields 
with newly extracted values.

**1b. Vision pass for low-confidence pages**

After the text extraction, check if the extracted text for any page segment is 
shorter than 200 characters (indicating a visual-heavy or table-heavy page that 
PyMuPDF read poorly).

For each such low-confidence segment (max 3 pages to control cost):
- Render that page as an image via PyMuPDF at 200 DPI (reuse the pattern already 
  in services/document_processor.py)
- Send it to GPT-4o Vision with prompt:
  "This is a page from a Brazilian financial document. List ALL financial products 
   mentioned: tickers, fund names, structured product names. Also extract any key 
   data visible in tables or charts (returns, ratings, terms, prices). 
   Return JSON: {products_found: [{ticker, name, type}], key_data_found: string}"
- Merge any newly found products into the identification result before the 
  cascade match step

**1c. Auto-detection of no-product materials**

After both extraction steps, if zero products are identified with confidence > 0.4, 
add a field to the response: "no_products_detected": true, along with a 
"material_nature_guess": one of ["educacional", "macro", "regulatorio", "outro"].

The frontend will use this flag (see Change 2c).
