# 🚀 Guia de Deploy Replit → GitHub → Railway

## ⚠️ PROBLEMAS QUE ESTE GUIA PREVINE

1. **Arquivos compilados não sobem para o GitHub** (dist/, build/ no .gitignore)
2. **Código legado/antigo misturado com código atual**
3. **Dessincronização entre Replit e GitHub**
4. **Perda de tempo e dinheiro com deploys quebrados**

---

## 📋 CHECKLIST PRÉ-DEPLOY (OBRIGATÓRIO)

### ✅ ANTES DE COMMITAR QUALQUER COISA

- [ ] **LIMPAR CÓDIGO LEGADO**: Deletar TODAS as pastas/arquivos não utilizados
- [ ] **VERIFICAR .gitignore**: Garantir que arquivos essenciais NÃO estão sendo ignorados
- [ ] **BUILD LOCAL**: Compilar o projeto no Replit e verificar se funciona
- [ ] **TESTAR LOCALMENTE**: Rodar o projeto completo no Replit antes de subir

---

## 🧹 PASSO 1: LIMPEZA DE CÓDIGO LEGADO

### O que fazer ANTES de qualquer commit:

```bash
# 1. Liste TODOS os arquivos e pastas do projeto
ls -la

# 2. Identifique o que NÃO está sendo usado:
# - Pastas antigas (ex: old/, backup/, deprecated/)
# - Arquivos duplicados
# - Código comentado em massa
# - Dependências não utilizadas

# 3. DELETE SEM DÓ:
rm -rf pasta_antiga/
rm arquivo_legado.js

# 4. Limpe o package.json:
# - Remova dependências não utilizadas
# - Mantenha apenas o necessário
```

### ⚠️ REGRA DE OURO:
**Se não está sendo usado AGORA, DELETE. Não guarde "por precaução".**

O Git guarda todo o histórico. Se precisar, você recupera depois.

---

## 📦 PASSO 2: CONFIGURAR .gitignore CORRETAMENTE

### ⚠️ ATENÇÃO CRÍTICA: Build Files

Existem **DOIS CENÁRIOS** diferentes:

### 🔴 CENÁRIO 1: Deploy com Docker/Railway (SEU CASO)

**O build acontece NO SERVIDOR, não localmente.**

`.gitignore` deve incluir:
```gitignore
# Dependencies
node_modules/
npm-debug.log
yarn-error.log

# Environment
.env
.env.local
.env.production

# Replit
.replit
replit.nix
.config/

# Build artifacts (serão gerados no servidor)
dist/
build/
.next/
out/

# Cache
.cache/
.vite/
.turbo/

# OS
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/
*.swp
*.swo
```

**POR QUE?**
- O Dockerfile faz `npm run build` no servidor
- Subir `dist/` local pode causar conflitos
- Build local pode ter caminhos errados

### 🟢 CENÁRIO 2: Deploy estático (GitHub Pages, Netlify)

**O build acontece LOCALMENTE, arquivos vão para o repo.**

`.gitignore` NÃO deve incluir `dist/` ou `build/`:
```gitignore
# Dependencies
node_modules/
npm-debug.log

# Environment
.env
.env.local

# NÃO ignore dist/ ou build/ neste caso!
```

## 🔍 PASSO 3: VERIFICAÇÃO ANTES DO COMMIT

### Execute estes comandos NO REPLIT:

```bash
# 1. Verifique o que será commitado
git status

# 2. Verifique o que está sendo IGNORADO
git status --ignored

# 3. ATENÇÃO: Se você está usando Docker/Railway, dist/ DEVE aparecer como ignored
# Se não aparecer, seu .gitignore está errado!

# 4. Veja o diff do que será enviado
git diff

# 5. Liste arquivos trackeados
git ls-files
```

### ⚠️ VERIFICAÇÕES CRÍTICAS:

```bash
# Certifique-se que NÃO existem:
git ls-files | grep -E "old|backup|deprecated|legacy|test_old"

# Se retornar algo, você tem código legado sendo trackeado!
# Delete esses arquivos/pastas ANTES de commitar
```

## 🐳 PASSO 5: CONFIGURAR DOCKER CORRETAMENTE

### Dockerfile para React + Node.js:

```dockerfile
# Build stage para o frontend
FROM node:18-alpine AS frontend-builder
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build

# Build stage para o backend
FROM node:18-alpine AS backend-builder
WORKDIR /app/backend
COPY backend/package*.json ./
RUN npm ci
COPY backend/ ./

# Production stage
FROM node:18-alpine
WORKDIR /app

# Copiar build do frontend
COPY --from=frontend-builder /app/frontend/dist ./frontend/dist

# Copiar backend
COPY --from=backend-builder /app/backend ./backend

# Instalar apenas dependências de produção
WORKDIR /app/backend
RUN npm ci --only=production

EXPOSE 3000
CMD ["node", "src/index.js"]
```

### ⚠️ VERIFICAÇÃO DO DOCKERFILE:

```bash
# Teste o build Docker localmente NO REPLIT (se possível):
docker build -t test-app .

# Se falhar, corrija ANTES de fazer commit
```

## 🔄 PASSO 6: WORKFLOW DE DEPLOY CORRETO

### Processo completo (siga NA ORDEM):

```bash
# 1. LIMPAR código legado
rm -rf old_code/ backup/ *.old.*

# 2. TESTAR build local
cd frontend
npm run build
cd ..

# 3. VERIFICAR .gitignore
cat .gitignore
# Confirme que dist/ está lá

# 4. VERIFICAR status
git status
git status --ignored

# 5. ADD apenas o necessário
git add .

# 6. VERIFICAR o que será commitado
git status
# dist/ NÃO deve aparecer em "Changes to be committed"

# 7. COMMIT com mensagem clara
git commit -m "feat: deploy ready - código limpo, sem legado"

# 8. PUSH para GitHub
git push origin main

# 9. Deploy no Railway
# Railway vai clonar o repo e fazer o build fresh
```

## 🚨 ERROS COMUNS E SOLUÇÕES

### Erro 1: "dist/ não está no GitHub mas deveria estar"

**CAUSA**: Você está fazendo deploy estático, não Docker.

**SOLUÇÃO**:
```bash
# Remova dist/ do .gitignore
nano .gitignore
# Delete a linha "dist/"

# Force add
git add -f frontend/dist/
git commit -m "add: dist files for static deploy"
git push
```

### Erro 2: "Código legado foi para o GitHub"

**CAUSA**: Não limpou antes de commitar.

**SOLUÇÃO**:
```bash
# Delete os arquivos
rm -rf codigo_legado/

# Remove do Git
git rm -r codigo_legado/
git commit -m "clean: remove legacy code"
git push
```

### Erro 3: "Build falha no Railway mas funciona no Replit"

**CAUSA**: Dependências ou env vars diferentes.

**SOLUÇÃO**:
```bash
# 1. Verifique package.json tem todos os scripts:
{
  "scripts": {
    "build": "vite build",
    "start": "node src/index.js"
  }
}

# 2. Configure env vars no Railway exatamente como no Replit

# 3. Verifique Dockerfile copia tudo necessário
```

## 🎯 REGRAS DE OURO

### 1. **UM CÓDIGO, UMA FONTE DE VERDADE**
- Não mantenha múltiplas versões do mesmo arquivo
- Não crie pastas "old", "backup", "test"
- Use Git branches para experimentação

### 2. **BUILD NO LUGAR CERTO**
- Deploy Docker? Build no servidor (dist/ no .gitignore)
- Deploy estático? Build local (dist/ NO repo)

### 3. **SEMPRE TESTE ANTES**
- Build local no Replit
- Verifique git status
- Confirme .gitignore

### 4. **LIMPE ANTES DE COMMITAR**
- Delete código não usado
- Remova comentários grandes
- Limpe console.logs

### 5. **MENSAGENS DE COMMIT CLARAS**
```bash
✅ git commit -m "feat: add user authentication"
✅ git commit -m "fix: resolve build error in production"
✅ git commit -m "clean: remove deprecated API calls"

❌ git commit -m "update"
❌ git commit -m "changes"
❌ git commit -m "fix"
```

## 🔧 COMANDOS ÚTEIS DE EMERGÊNCIA

### Se já fez commit errado:

```bash
# Desfazer último commit (mantém alterações)
git reset --soft HEAD~1

# Desfazer último commit (descarta alterações)
git reset --hard HEAD~1

# Remover arquivo do staging
git restore --staged arquivo.js

# Remover arquivo do Git mas manter no disco
git rm --cached arquivo.js
```

### Se precisa forçar limpeza:

```bash
# Ver todos os arquivos trackeados
git ls-files

# Remover tudo que está no .gitignore mas foi commitado
git rm -r --cached .
git add .
git commit -m "clean: apply .gitignore retroactively"
```

## 🎓 RESUMO EXECUTIVO

**ANTES DE CADA DEPLOY:**

1. Delete código legado
2. Configure .gitignore pro SEU cenário
3. Verifique git status
4. Teste build local
5. Commit apenas o necessário
6. Push
7. Verifique deploy

**LEMBRE-SE:**
- Git guarda histórico. Delete sem medo.
- Menos código = menos bugs
- Build no servidor > build local (Docker)
- Mensagens claras salvam tempo depois

## ❓ DÚVIDAS FREQUENTES

**Q: E se eu precisar do código antigo depois?**
A: Git guarda tudo. Use `git checkout <commit-hash>` para recuperar.

**Q: Quanto do build devo versionar?**
A: Se usa Docker: NADA (dist/ no .gitignore). Se é estático: TUDO (dist/ no repo).

**Q: Posso ter package-lock.json no .gitignore?**
A: NÃO! Sempre versione package-lock.json ou yarn.lock.
