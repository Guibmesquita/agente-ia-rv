## PHASE 1 — KNOWLEDGE BASE RECONNAISSANCE

### 1.1 — Inventory the existing content

Query the database:

```sql
-- Material overview
SELECT 
  m.id,
  m.name,
  m.material_type,
  m.publish_status,
  m.is_indexed,
  m.is_committee_active,
  COUNT(cb.id) as block_count,
  COUNT(de.doc_id) as embedding_count,
  array_agg(DISTINCT p.ticker) as linked_tickers
FROM materials m
LEFT JOIN content_blocks cb ON cb.material_id = m.id
LEFT JOIN document_embeddings de ON de.material_id::int = m.id
LEFT JOIN material_product_links mpl ON mpl.material_id = m.id
LEFT JOIN products p ON p.id = mpl.product_id
WHERE m.publish_status = 'publicado' AND m.is_indexed = true
GROUP BY m.id, m.name, m.material_type, m.publish_status, 
         m.is_indexed, m.is_committee_active
ORDER BY block_count DESC;
```

```sql
-- ContentBlock type distribution
SELECT block_type, COUNT(*) as count, 
       AVG(LENGTH(content)) as avg_content_length,
       MIN(LENGTH(content)) as min_length,
       MAX(LENGTH(content)) as max_length
FROM content_blocks
GROUP BY block_type;
```

```sql
-- Embedding metadata completeness
SELECT 
  COUNT(*) as total_embeddings,
  COUNT(CASE WHEN product_ticker IS NOT NULL AND product_ticker != '' THEN 1 END) as with_ticker,
  COUNT(CASE WHEN gestora IS NOT NULL AND gestora != '' THEN 1 END) as with_gestora,
  COUNT(CASE WHEN valid_until IS NOT NULL THEN 1 END) as with_expiry,
  AVG(LENGTH(content)) as avg_chunk_length
FROM document_embeddings
WHERE publish_status = 'publicado';
```

```sql
-- Products inventory
SELECT id, name, ticker, product_type, 
       (key_info IS NOT NULL AND key_info != '{}' AND key_info != 'null') as has_key_info,
       jsonb_array_length(COALESCE(name_aliases::jsonb, '[]'::jsonb)) as alias_count
FROM products
WHERE status = 'ativo'
ORDER BY product_type, ticker;
```

## PHASE 2 — RETRIEVAL QUALITY TESTS

For each test below, do NOT call the WhatsApp agent. Instead, call the 
EnhancedSearch pipeline directly (instantiate EnhancedSearch and call .search()) 
to isolate retrieval quality from generation quality.

Write a test script: audit/test_retrieval.py

### 2.1 — Direct fact retrieval tests (15 tests minimum)

For each fact in the Ground Truth Fact Bank, formulate 3 query variants:
- Variant A: Direct/explicit (ex: "qual o dividend yield do BTLG11?")
- Variant B: Natural language (ex: "BTLG11 tá pagando bem? quanto de DY?")  
- Variant C: Implicit/contextual (ex: "esse fundo de logística vale a pena?")
  — for Variant C, pre-load the conversation context with the ticker so 
    ConversationContextManager has it

For each query, record:
- Did the chunk containing the ground truth fact appear in results? (yes/no)
- If yes: what rank position? (1st, 2nd... 8th)
- Composite score of the ground truth chunk
- composite score of top-ranked chunk
- similarity_threshold: did the ground truth chunk pass it?
- Which retrieval layer found it: EntityResolver (layer 0), multi-query vector, 
  direct ticker search, fuzzy fallback, or not found at all

Calculate and report:
- Recall@3: % of ground truth facts found in top 3 results
- Recall@6: % found in top 6
- MRR (Mean Reciprocal Rank): average of 1/rank for found facts
- Layer distribution: which layer is doing most of the work?

### 2.2 — Chunking quality tests

For each material in the test set, run these diagnostics:

**Fragmentation test:** 
For a fact that spans multiple sentences (or a table row + its header), 
check if it was split across multiple chunks. If so, do both chunks score 
high together, or does only one appear in results?

**Chunk length distribution:**
Plot (or list) the character lengths of all chunks for the material.
Flag chunks shorter than 100 chars (likely fragmentation artifacts) and 
longer than 2000 chars (likely too broad for precise retrieval).

**Table block quality test:**
For each tabela-type block, verify:
- Is the JSON structure complete (has headers and rows)?
- Is the converted Markdown representation readable and correct?
- Run a query that should retrieve a specific cell value from this table.
  Does the retrieval system find it?

### 2.3 — Entity resolution tests

Test EntityResolver specifically with these query types:
- Exact ticker: "BTLG11" → should resolve with confidence 1.0
- Informal name: "TG Core" or "TG Genial" (aliases) → should resolve via aliases
- Partial ticker: "PETR" → should resolve PETR3 and PETR4 via prefix match
- Gestora name: "Bresco" or whichever gestoras exist → should resolve products
- Unknown product: "XPTO99" → should return empty cleanly (no false positives)
- Ambiguous term: "fundo" (generic) → should NOT create false matches
- Multi-entity: "compare BTLG11 com HGLG11" → should resolve both

For each test, record:
- Resolved products (list)
- Confidence scores
- Match type (exact/ilike/prefix/name/alias)
- False positives if any
- Time taken (ms)

### 2.4 — Semantic boundary tests (SPECIALIST C adversarial battery)

These tests are designed to find failure modes:

**Cross-contamination test:**
Ask about Product A using the gestora of Product B. Example: 
"Como é a gestão da [gestora_of_B] no [ticker_A]?"
Expected: results should be from ticker_A, not ticker_B
Failure: chunks from ticker_B appear due to gestora match dominating scoring

**Temporal confusion test:**
If materials have valid_from/valid_until dates, ask:
"Qual o dividend yield atual do [ticker]?" for a product with an expired material
Expected: expired chunks filtered out
Failure: expired chunks appear with stale data

**Negative knowledge test:**
Ask about a product that EXISTS in the products table but has NO content blocks 
(no material linked or no embeddings). Example: create this condition if it 
doesn't exist naturally.
Expected: agent acknowledges limited information
Failure: agent fabricates or returns wrong product's data

**Committee isolation test:**
If any material has is_committee_active=True AND has a MaterialProductLink with 
excluded_from_committee=True for some product:
- Query that excluded product
- Expected: comite_tag = [NÃO-COMITÊ] on those chunks
- Failure: chunks tagged [COMITÊ] despite exclusion

**Threshold sensitivity test:**
For the similarity_threshold currently set (likely 0.8):
- Find the ground truth chunk with the LOWEST composite score (hardest to retrieve)
- Record its actual vector similarity distance
- If it falls below threshold: it's being silently dropped — that's a critical gap

### 2.5 — Query rewriting quality test

If services/query_rewriter.py exists and is active:
For each of the 3 query variants per fact (from 2.1), record:
- Original query
- Rewritten query(ies) produced
- Whether rewriting improved or degraded retrieval rank

## PHASE 6 — CRITICAL ANALYSIS AND IMPROVEMENT REPORT

This is the most important phase. After completing all tests above, each 
specialist produces their critical assessment.

### SPECIALIST A (RAG Engineer) must evaluate and score (1-10):

For each item, give a score AND a specific finding from the tests:

1. **Chunking strategy** — Are chunks the right size? Are logical units 
   (a table, a thesis paragraph, a risk section) kept together or fragmented?

2. **Embedding input quality** — Is the metadata prefix (Produto/Gestora/Tipo) 
   adding signal or noise? Is the content being embedded the best representation 
   of what the chunk is about?

3. **Retrieval completeness** — What is the Recall@6 across all tests? 
   Are facts being dropped due to threshold? Are some block types systematically 
   under-retrieved?

4. **Score composition** — Are the 6-factor weights optimal for this domain? 
   Is gestora_match (0.10) appropriate or should ticker_match dominate more? 
   Is recency_score (0.05) too low given that financial data ages quickly?

5. **Context assembly** — Are the right chunks being sent to GPT-4o? Is there 
   redundancy (same info in multiple chunks consuming context)? Is there 
   important context being cut (6-chunk limit may be too low for complex queries)?

6. **Reranking** — Does the system need a cross-encoder or LLM-based reranker 
   step between retrieval and generation?

7. **Query expansion** — Is multi-query expansion (up to 3 queries) generating 
   genuinely different semantic angles, or near-duplicates?

8. **Table retrieval** — Are tabela-type blocks being retrieved as effectively 
   as text blocks? Tables often contain the most precise financial data.

9. **Threshold calibration** — Is 0.8 similarity threshold too aggressive? 
   What % of ground truth facts fell below it?

10. **Missing capabilities** — What retrieval techniques exist in the literature 
    that this system lacks? (e.g., HyDE, parent-child chunking, metadata filtering 
    at query time, hybrid BM25+vector)

### SPECIALIST B (Financial Domain Expert) must evaluate and score (1-10):

1. **Yield/return data accuracy** — When asked about DY or expected return, 
   does the agent give the correct figure from the material?

2. **Investment thesis quality** — Does the agent capture the CORE thesis 
   or just surface-level information?

3. **Risk communication** — Are risks presented with appropriate specificity 
   or generically?

4. **Comparative analysis quality** — When comparing two products, does the 
   agent give substantive differentiation or vague statements?

5. **Committee language discipline** — Does the agent correctly use formal 
   recommendation language ONLY for [COMITÊ] products?

6. **Temporal awareness** — Does the agent flag when KB data may be stale and 
   proactively complement with web?

7. **Table data utilization** — When a specific number from a table is asked, 
   does the agent retrieve and present it precisely?

8. **Web data integration** — For live market data, is the information accurate, 
   current, and well-sourced?

9. **Practical advisor utility** — Would a financial advisor trust and use these 
   responses in a client conversation?

10. **Missing financial context** — What financial data or context that a good 
    advisor needs is consistently absent from responses?

**Failure mode catalog** — List every failure found with:
