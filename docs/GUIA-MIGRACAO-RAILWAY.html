Guia de Migração: Replit → Railway

Agente IA - RV  |  Documento para o usuário  |  Versão 1.0

1. Visão Geral

Este guia descreve o passo a passo que você (usuário/administrador) precisa executar para migrar a aplicação do Replit para o Railway. Tarefas de código já foram realizadas pelo agente — este documento foca em ações, validações e configurações que dependem de você.

O que muda para você?

AspectoAntes (Replit)Depois (Railway)

Cold startSim (delay ao acessar)Não — sempre online
Banco de dadosPostgreSQL do ReplitPostgreSQL no Railway
DeploysManual no ReplitAutomático via push no GitHub

2. Pré-requisitos

  • Conta no Railway (railway.com) — plano Pro recomendado
  • Conta no GitHub com acesso ao repositório do projeto
  • Acesso ao Replit atual (para extrair dados e secrets)
  • Acesso ao painel Azure AD (Microsoft Entra ID) para atualizar redirect URIs do SSO
  • Acesso ao painel do Z-API para atualizar webhooks

3. Etapa 1 — Conectar GitHub ao Railway

3.1 Verificar o repositório GitHub

  1. Acesse o repositório no GitHub e confirme que o código está atualizado.
  2. Verifique que os seguintes arquivos existem na raiz:
    • Dockerfile
    • railway.json
    • requirements.txt
    • .gitignore

3.2 Criar projeto no Railway

  1. Acesse railway.com e faça login.
  2. Clique em "New Project".
  3. Selecione "Deploy from GitHub Repo".
  4. Autorize o Railway a acessar sua conta GitHub (se ainda não fez).
  5. Selecione o repositório do projeto.
  6. O Railway detectará automaticamente o Dockerfile e o railway.json.
  • Projeto criado no Railway
  • Repositório GitHub conectado

4. Etapa 2 — Provisionar o Banco de Dados

4.1 Criar PostgreSQL no Railway

  1. No projeto Railway, clique em "+ New" → "Database" → "PostgreSQL".
  2. O Railway cria automaticamente e expõe a variável DATABASE_URL.

4.2 Habilitar extensão pgvector

  1. No painel do PostgreSQL no Railway, acesse a aba "Data" ou "Query".
  2. Execute: 
    CREATE EXTENSION IF NOT EXISTS vector;

4.3 Restaurar o backup do banco

  1. Baixe o arquivo backup_migracao.sql do Replit (disponível na raiz do projeto).
  2. No painel do Railway PostgreSQL, copie a connection string (formato: postgresql://user:pass@host:port/dbname).
  3. No seu terminal local (com psql instalado), execute: 
    psql "CONNECTION_STRING_DO_RAILWAY" < backup_migracao.sql
  4. Alternativamente, use uma ferramenta gráfica como DBeaver ou pgAdmin para importar.

4.4 Validar a restauração

Execute no console SQL do Railway:

SELECT count(*) FROM users;
SELECT count(*) FROM knowledge_chunks;
SELECT count(*) FROM conversations;

Compare os números com o ambiente Replit atual.

  • PostgreSQL criado no Railway
  • Extensão vector habilitada
  • Backup restaurado com sucesso
  • Contagem de registros confere

6. Etapa 4 — Deploy e Verificação Inicial

6.1 Disparar o primeiro deploy

  1. Após configurar as variáveis, o Railway fará o deploy automaticamente.
  2. Acompanhe os logs em tempo real na aba "Deployments".
  3. Aguarde a mensagem de que o servidor está rodando (porta ativa).

6.2 Testar o health check

Acesse no navegador:

https://SEU-DOMINIO.up.railway.app/health

Deve retornar {"status": "healthy", ...}.

  • Deploy concluído sem erros
  • Health check retorna status healthy

7. Etapa 5 — Reconfigurar Integrações Externas

7.1 SSO Microsoft (Azure AD / Entra ID)

  1. Acesse o Azure Portal → App Registrations → selecione o app.
  2. Vá em Authentication → Redirect URIs.
  3. Adicione a nova URI: 
    https://SEU-DOMINIO.up.railway.app/auth/callback
  4. Você pode manter a URI antiga do Replit enquanto estiver em transição.

7.2 Webhook do Z-API (WhatsApp)

  1. Acesse o painel do Z-API.
  2. Na instância configurada, atualize a URL do webhook para: 
    https://SEU-DOMINIO.up.railway.app/api/whatsapp/webhook
  3. Teste enviando uma mensagem pelo WhatsApp e verifique nos logs do Railway se chegou.

7.3 Domínio personalizado (opcional)

  1. No Railway, vá em Settings → Networking → Custom Domain.
  2. Adicione seu domínio (ex: agente.suaempresa.com.br).
  3. Configure o DNS (CNAME) conforme instruções do Railway.
  4. Se usar domínio personalizado, atualize ALLOWED_ORIGINS, APP_BASE_URL, redirect URI do Azure AD e webhook do Z-API.
  • Redirect URI do SSO atualizada no Azure AD
  • Webhook do Z-API apontando para o novo domínio
  • Domínio personalizado configurado (se aplicável)

8. Etapa 6 — Validação Funcional Completa

Teste cada funcionalidade abaixo no novo ambiente. Marque quando validado:

8.1 Autenticação

  • Login via SSO Microsoft funciona
  • Sessão mantida após refresh da página
  • Logout funciona corretamente

8.2 Conversas / Chat

  • Histórico de conversas carrega corretamente
  • Nova conversa pode ser iniciada
  • Agente responde com contexto da base de conhecimento
  • Pesquisa semântica retorna resultados relevantes

8.3 Base de Conhecimento

  • Produtos listados corretamente
  • Upload de novos materiais funciona
  • Busca por produto/material funciona
  • Fila de revisão acessível

8.4 Insights (Dashboard)

  • Dashboard carrega com dados
  • Gráficos e KPIs exibidos
  • Filtros funcionam

8.5 Campanhas

  • Lista de campanhas carrega
  • Criação de nova campanha funciona

8.6 WhatsApp (Z-API)

  • Mensagem recebida aparece no sistema
  • Resposta do agente é enviada ao WhatsApp

8.7 Custos

  • Painel de custos carrega
  • Dados de consumo OpenAI exibidos

8.8 Configuração do Agente

  • Página de configuração do cérebro acessível
  • Alterações salvas com sucesso

9. Etapa 7 — Descomissionar o Replit

⚠ Importante: Só execute esta etapa após validar completamente o ambiente Railway (Etapa 6). Recomenda-se manter o Replit ativo por pelo menos 1 semana após a migração como fallback.
  1. Confirme que todas as funcionalidades foram validadas na Etapa 6.
  2. Remova a URI antiga do Replit no Azure AD (Redirect URIs).
  3. Atualize ALLOWED_ORIGINS removendo o domínio Replit.
  4. Pause ou delete o projeto no Replit.
  • Todas as validações da Etapa 6 passaram
  • Ambiente rodando estável por pelo menos 48h
  • Replit descomissionado

11. Troubleshooting

O deploy falha no Railway

  • Verifique os logs na aba "Deployments". Erros de build aparecem ali.
  • Confirme que todas as variáveis de ambiente obrigatórias estão definidas.
  • Certifique-se de que o DATABASE_URL do PostgreSQL Railway está linkado ao serviço.

Health check falha

  • Acesse /health manualmente. Se retornar erro, verifique os logs.
  • Confirme que o banco de dados está acessível (variável DATABASE_URL correta).

SSO não funciona

  • Verifique se a redirect URI no Azure AD bate exatamente com a URL do Railway (incluindo https://).
  • Confirme que MICROSOFT_CLIENT_ID, MICROSOFT_CLIENT_SECRET e MICROSOFT_TENANT_ID estão configurados.

WhatsApp não recebe/envia mensagens

  • Verifique se o webhook do Z-API aponta para a URL correta do Railway.
  • Confirme que ZAPI_INSTANCE_ID, ZAPI_TOKEN e ZAPI_CLIENT_TOKEN estão configurados.
  • Teste o endpoint diretamente: POST /api/whatsapp/webhook.

Dados não aparecem

  • Confirme que o backup foi restaurado corretamente (Etapa 2.4).
  • Verifique se a extensão vector está habilitada: SELECT * FROM pg_extension WHERE extname = 'vector';
